# Incident Postmortem: Duplicate Volunteer Shift Assignments During Batch Import

## Summary

On 2026-06-10, a batch import process used by our disaster-response nonprofit created duplicate volunteer shift assignments in production. The import reprocessed a set of already-imported rows without a reliable idempotency check, resulting in some volunteers being assigned to the same shift more than once. The issue was detected the same day by a program coordinator reviewing shift rosters. We disabled the import job, corrected the affected records, and notified impacted coordinators. No volunteer was ultimately turned away from an active disaster response event, but the incident created confusion, increased manual work, and temporarily reduced trust in the scheduling data.

## Impact

- 312 shift assignment records were duplicated across 47 volunteer shift slots.
- 28 volunteers received duplicate assignments for the same shift.
- Coordinators spent approximately 6 person-hours identifying and reconciling records.
- Two shift leads reported uncertainty about final staffing counts during live operations.
- No safety incident occurred, but the error increased operational risk during a time-sensitive response period.

## Timeline

- **09:05** — Batch import began for a weekly roster update from a partner spreadsheet export.
- **09:12** — Import completed successfully and reported no errors.
- **10:40** — A program coordinator noticed duplicate names in a shift roster.
- **10:55** — Engineering and operations were notified.
- **11:10** — The import job was disabled to prevent further duplication.
- **11:35** — Logs and source files were reviewed; duplicate inserts were confirmed.
- **12:20** — A script was run to identify and mark duplicated assignment records.
- **13:05** — Coordinators received corrected roster exports.
- **15:00** — The import pipeline was restored with a temporary manual review step.
- **Next day** — A follow-up audit confirmed no additional affected shifts.

## Contributing Factors

1. **Missing idempotency in the import pipeline**  
   The job inserted shift assignments based on row content, but did not enforce a stable external key or deduplication constraint at the application or database layer.

2. **Repeat processing of unchanged input**  
   The import source produced an identical export with the same assignments, and the pipeline treated the file as new because it lacked file fingerprint tracking.

3. **Insufficient validation and alerts**  
   The job validated row format and required fields, but not uniqueness of volunteer-to-shift assignments. Duplicate creation did not trigger any alert.

4. **Operational dependence on batch imports**  
   Coordinators rely heavily on the import job for rapid schedule updates during disaster response, which reduced the opportunity for manual cross-checking before assignments were used.

## Root Cause Analysis

The root cause was a design gap in the batch import workflow: it lacked a reliable mechanism to identify whether an assignment had already been imported. The system accepted rows as independent records and created new shift assignments even when the volunteer, shift, and source data matched an existing assignment.

This was not caused by a single code defect, but by an incomplete data model and insufficient safeguards around repeated imports. The job was built to prioritize successful ingestion of partner-provided schedules, but it did not include idempotent writes, uniqueness constraints, or replay protection. As a result, reprocessing the same data produced duplicate records.

## What Went Well

- The issue was detected quickly by an attentive coordinator during routine roster review.
- The import job could be disabled immediately, limiting further duplication.
- Logs and source files were sufficient to reconstruct the failure mode.
- The team was able to correct records and redistribute accurate rosters within hours.
- Cross-functional communication between operations and engineering was prompt and clear.

## What Went Poorly

- We relied on a single import path without strong guardrails against duplicate writes.
- The system reported the job as successful despite creating invalid data.
- There was no automated monitoring for duplicate assignment anomalies.
- Corrective work required manual investigation during an already high-pressure operational period.
- The lack of an explicit reconciliation process delayed confidence in the roster data.

## Action Items

1. **Add database uniqueness constraints** for volunteer, shift, and source-assignment identifiers to prevent duplicate writes.
2. **Implement idempotency keys** in the import pipeline so repeated processing of the same file does not create new records.
3. **Track file fingerprints and import history** to detect and skip unchanged partner exports.
4. **Add post-import validation checks** that compare assignment counts and flag duplicate rates above zero.
5. **Create alerting for anomalous import outcomes**, including repeated inserts for the same shift/volunteer pair.
6. **Introduce a reconciliation report** for coordinators showing newly added, updated, and rejected assignments after each import.
7. **Document a manual fallback procedure** for verifying roster integrity before assignments are used in active response operations.

## Closing

This incident exposed a gap between successful ingestion and correct data handling. We have already applied temporary controls and are implementing durable protections so that batch imports remain safe, repeatable, and trustworthy for mission-critical volunteer coordination.
