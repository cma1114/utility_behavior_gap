# Postmortem: Duplicate Volunteer Shift Assignments from Batch Import

## Summary

On 2026-06-12, a batch import of volunteer shift assignments for the disaster-response nonprofit created duplicate shift records for a subset of volunteers. The issue affected scheduling accuracy for one deployment planning cycle and required manual cleanup by operations staff. No volunteer was directly assigned to conflicting physical shifts on-site, but several volunteers received duplicate notifications and the scheduling team had to spend significant time reconciling records before assignments could be trusted again.

This was a process and system failure, not an individual failure. The import workflow lacked sufficient safeguards against reprocessing the same data, and the data model did not prevent duplicate assignment creation under repeated submissions.

## Impact

- Duplicate shift assignments were created for approximately 140 volunteer records across 3 deployment regions.
- About 90 volunteers received duplicate confirmation emails.
- Scheduling staff spent roughly 6 hours identifying and removing duplicates and verifying the clean state of the roster.
- A small number of volunteers were confused about whether they were double-booked, resulting in support emails and follow-up calls.
- No disaster response coverage was lost, but the team’s confidence in the scheduling system was reduced during a time-sensitive planning period.

## Timeline

- **09:10** — Operations team uploaded a CSV batch containing volunteer shift assignments.
- **09:12** — Import job completed successfully and posted a “finished” notification.
- **09:40** — Support inbox received the first volunteer complaint about duplicate assignment emails.
- **10:05** — Scheduling staff noticed repeated names in the assignment dashboard.
- **10:20** — Team paused additional imports and began manual review.
- **11:00** — Investigation identified that the same batch file had been processed twice after a retry from the import service.
- **11:30** — Engineers confirmed that duplicate creation was possible because the import endpoint was not idempotent.
- **12:15** — A cleanup script removed duplicate assignment records and suppressed further duplicate notifications.
- **13:00** — Roster verification completed and scheduling resumed.

## Root Cause Analysis

The immediate cause was a repeated execution of the batch import job against the same input file. The system treated the second submission as a new request and created a second assignment record for each matched volunteer-shift pair.

The underlying root cause was the lack of idempotency and duplicate protection in the import path. Specifically:

1. The import API accepted batch files without a stable import identifier.
2. The processing pipeline did not check whether a volunteer-shift assignment already existed before inserting a new record.
3. Retry behavior in the import service was not designed to distinguish between a failed request and a request that had already succeeded.
4. The user interface presented the successful completion message before post-processing verification had completed, making the import appear safe even when downstream duplication was still possible.

## Contributing Factors

- The batch import feature was originally designed for low volume and was not hardened for retries and re-submissions.
- There was no database uniqueness constraint on the combination of volunteer ID, shift ID, and import period.
- The job runner retried on transient network errors without an idempotency key or deduplication token.
- Monitoring focused on job completion, not record-level invariants such as “one volunteer, one shift assignment.”
- Operational practice relied on manual spot checks rather than automated reconciliation after imports.

## What Went Well

- The issue was detected quickly, within about 30 minutes of import completion.
- Scheduling staff responded promptly and paused further imports before the problem spread.
- Logs and job metadata were sufficient to confirm the failure mode without prolonged investigation.
- The team produced a cleanup script quickly and restored data integrity the same day.

## What Went Poorly

- The system allowed duplicate assignment creation at all.
- The import workflow gave a false sense of safety by reporting success before full verification.
- Duplicate notifications reached volunteers, creating confusion during an already stressful response period.
- There was no alert for duplicate record growth, so the issue was found by users rather than by automation.
- The team had to rely on manual cleanup, which consumed time that should have been spent on mission work.

## Action Items

1. **Add idempotency to batch imports**  
   Introduce a stable import request ID and ensure repeated submissions of the same batch cannot create duplicate assignments.

2. **Enforce database uniqueness**  
   Add a unique constraint on volunteer ID + shift ID + scheduling period, with clear handling for conflicts.

3. **Make retries safe**  
   Update the job runner so retries re-use the same import token and do not re-execute record creation blindly.

4. **Improve validation and reconciliation**  
   Add automated post-import checks that compare expected versus actual assignment counts and flag duplicates immediately.

5. **Delay success reporting until verification completes**  
   Change the UI and job status flow so imports are only marked successful after record integrity checks pass.

6. **Add duplicate-growth monitoring**  
   Create an alert for unexpected increases in duplicate assignment records or repeated notifications.

7. **Document operational safeguards**  
   Update the import runbook with pre-upload checks, rollback steps, and escalation guidance for scheduling staff.

8. **Test failure and retry cases**  
   Add integration tests covering duplicate submissions, transient failures, and partial retries.

This incident showed that reliability work for administrative workflows is as important as reliability work for core public-facing systems. The next step is to make batch imports safe by default and detectable when they are not.
