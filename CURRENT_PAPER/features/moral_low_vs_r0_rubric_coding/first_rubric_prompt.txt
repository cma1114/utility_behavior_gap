## Summary

For approximately two hours, the clinic portal displayed stale medication refill statuses to some users. Patients and staff saw outdated information indicating that refill requests were still pending or had not been processed when, in fact, the underlying refill state had changed. - Duration: ~2 hours

- **09:50** — Support tickets and internal alerts confirm the issue is reproducible.

- **10:20** — Mitigation applied by disabling cache reuse for refill status responses.
- **10:40** — Portal reflects current refill statuses again after cache expiry and refresh.
- **11:10** — Incident declared resolved after validation across several user accounts and refill states.

The root cause was a bug in the cache invalidation logic for medication refill status responses. The portal uses cached status data to reduce repeated reads from the underlying service. When a refill request changed state, the system was supposed to invalidate the corresponding cache entry so the next read would fetch fresh data. In the affected code path, invalidation was only triggered for one of several state-transition events. As a result, some refill statuses remained cached after updates and were served to users until the cache entry naturally expired.

This was not a single-point operator error. The bug arose from a mismatch between the expected data lifecycle and the implementation of cache invalidation logic, combined with insufficient test coverage for state transitions that occur outside the most common path.

- The cache invalidation logic was distributed across multiple code paths, making it easy to miss a transition.

- The portal UI did not surface freshness metadata, so users had no indication that displayed data might be outdated.

## What Went Well

4. **Shorten cache TTL for critical status fields**

6. **Document portal data consistency expectations**

For approximately two hours, the clinic portal displayed stale medication refill statuses to some users due to a cache invalidation bug. The issue affected the accuracy of refill status information shown to clinic staff, creating confusion and requiring manual verification in some cases. No evidence suggests that incorrect data was written back to source systems, and no patient safety event was identified. The issue was resolved after the cache behavior was corrected and affected entries were refreshed.

## Impact

Users viewing medication refill statuses in the clinic portal saw outdated information for up to two hours after underlying status changes occurred. This impacted workflow efficiency for clinic staff, who relied on the portal to confirm refill progress. Some users had to cross-check with downstream systems or contact support to verify the current status.

Estimated impact:
- Duration: ~2 hours
- Scope: Limited subset of portal users and medication refill status views
- Severity: Moderate operational impact; no confirmed patient harm
- Data integrity: Source-of-truth systems remained correct; the portal display was stale

- **T0** — Medication refill status changes began occurring in the source system as expected.
- **T0 + minutes** — The clinic portal continued serving cached refill status values instead of refreshed values.

- **T0 + 120 minutes** — Investigation concluded and monitoring confirmed recovery.

## Root Cause Analysis

The root cause was a cache invalidation bug in the clinic portal’s medication refill status retrieval path. When refill status values changed in the source system, the portal’s cache did not reliably invalidate the corresponding cached entry. As a result, users continued to see an older status until the cache entry expired naturally or was refreshed manually.

This bug was introduced during a prior optimization intended to reduce load on the backend status service. The invalidation path depended on a status-transition event that was not consistently emitted for all update scenarios. When that event was missing, the cached value remained valid from the portal’s perspective, even though it was no longer current in the source of truth.

- **Event-dependent invalidation:** Cache freshness relied on a status-change event rather than direct reconciliation with the source system.
- **Insufficient test coverage:** Automated tests did not include scenarios where refill statuses changed through alternate update paths.
- **Lack of cache staleness monitoring:** We did not have alerts specifically detecting divergence between cached status values and source-system values.
- **Limited visibility:** The portal did not expose cache age or freshness metadata, making stale data harder to identify quickly.
- **Optimization tradeoff:** The performance improvement reduced backend calls but introduced a stronger dependence on accurate invalidation semantics.

## What Went Well

- Users quickly reported the discrepancy, helping us detect the issue faster than internal monitoring alone would have.
- Engineering was able to identify the cache invalidation gap and deploy a mitigation without requiring a full rollback.
- The underlying source-of-truth systems remained correct, limiting the incident to display-layer inconsistency.
- Communication between support, engineering, and operations was prompt and collaborative.

## What Went Poorly

- The cache invalidation logic did not account for all refill status transition paths.
- We lacked automated validation that cached statuses matched source data after updates.
- Monitoring did not surface the problem until users noticed it.
- The impact assessment took longer than desired because we had limited visibility into how many records were affected and for how long.
- The change that introduced the issue did not include a sufficiently robust review of failure modes related to stale data.
