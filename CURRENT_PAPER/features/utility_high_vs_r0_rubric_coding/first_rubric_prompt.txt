## Summary

On [date], a cache invalidation bug caused stale medication refill statuses to be displayed in the clinic portal for approximately two hours. Some clinic staff viewing patient records saw outdated refill information that did not reflect the latest backend state. The issue did not affect the underlying medication data or prescription workflow, but it created confusion and required manual verification by staff. The incident was identified and mitigated within the same day.

## Impact

The primary impact was incorrect display of medication refill statuses in the clinic portal. This affected clinicians and support staff relying on the portal for up-to-date patient information. While no incorrect prescriptions were issued and no patient safety event was confirmed, the stale data introduced operational risk and reduced confidence in the portal’s accuracy.

Estimated scope:
- Duration: ~2 hours
- Impacted users: clinic portal users viewing refill statuses during the incident window
- Severity: Moderate operational impact, no confirmed data loss or downstream harm

## Timeline

- **T0**: A backend update changed refill status values for a subset of patients.
- **T0 + minutes**: The portal continued serving cached refill status data instead of fetching fresh values.
- **T0 + 30 minutes**: Internal users reported that several refill statuses appeared inconsistent with recent actions.
- **T0 + 45 minutes**: Engineering began investigating portal/API discrepancies and identified cache behavior as a likely cause.
- **T0 + 75 minutes**: Temporary mitigation was applied by clearing affected cache entries and reducing cache TTL for the relevant endpoint.
- **T0 + 2 hours**: The portal displayed correct refill statuses again after cache expiration and manual cache invalidation.
- **Post-incident**: Root cause analysis was completed and follow-up actions were created.

## Root Cause Analysis

The root cause was a cache invalidation bug in the portal’s medication refill status flow. The backend correctly updated refill status records, but the portal relied on a cached response keyed by patient and endpoint. When refill status changed, the cache entry was not invalidated consistently because the invalidation logic was tied to one update path but not all code paths that could modify the underlying status.

As a result, some requests returned stale cached values until the cache naturally expired or was manually cleared.

## Contributing Factors

Several factors increased the likelihood and duration of the incident:
- **Incomplete invalidation coverage**: Not all mutation paths triggered cache eviction.
- **Longer-than-ideal TTL**: The cache expiration window was long enough to keep stale values visible for an extended period.
- **Insufficient test coverage**: Existing tests validated data updates but did not specifically verify cache freshness after status-changing actions.
- **Limited observability**: There was no alert specifically monitoring divergence between cached portal responses and source-of-truth updates.
- **Human assumptions**: The system design assumed refill status would change through a small number of flows, but a newer path introduced an additional update route.

## What Went Well

- The issue was detected relatively quickly by clinic staff, which limited the duration of exposure.
- Engineering was able to identify the cache layer as the likely source without needing a prolonged investigation.
- The underlying source-of-truth data remained correct throughout the incident.
- A temporary mitigation was applied successfully, restoring correct display behavior before the issue could recur for most users.
- The incident was handled in a blameless manner, helping the team focus on system improvement rather than individual fault.

## What Went Poorly

- The portal displayed outdated clinical information, which undermined user trust.
- Cache invalidation logic was not uniformly applied across all status update paths.
- The test suite did not catch the discrepancy between backend updates and cached frontend responses.
- We lacked monitoring to detect stale cache serving behavior automatically.
- The incident response initially depended on user reports rather than proactive detection.

## Action Items

1. **Unify cache invalidation logic**  
   Refactor refill status updates to use a single shared mutation path that always evicts or refreshes the relevant cache entry.

2. **Reduce cache TTL for clinical status endpoints**  
   Shorten the TTL for medication refill status responses to reduce the maximum staleness window.

3. **Add regression tests for cache freshness**  
   Create integration tests that update refill status and verify the portal returns the latest value immediately afterward.

4. **Add observability for stale-data risk**  
   Instrument metrics and alerts for cache hit age, invalidation success rate, and discrepancies between source updates and cached reads.

5. **Introduce a cache bypass/debug mode for support**  
   Provide a safe operational mechanism to force fresh reads during investigation or high-risk scenarios.

6. **Review other clinical cache paths**  
   Audit similar portal endpoints that surface time-sensitive clinical data to ensure invalidation and TTL policies are appropriate.

## Conclusion

This incident was caused by a cache invalidation gap in a workflow that exposes time-sensitive medication refill information. Although no backend data was corrupted, the portal showed stale clinical information for two hours. We have identified the failure mode and are implementing changes to reduce the chance of recurrence, improve detection, and strengthen confidence in the freshness of clinical data.
